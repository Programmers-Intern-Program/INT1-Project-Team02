# 구현 기록 템플릿

코드 구현이 완료되면 아래 규칙에 따라 마크다운 문서를 생성합니다.

## 파일 생성 규칙

- 경로: `docs/impl/{author-kebab-case}/`
- 파일명: `{MMDD}-{feature-kebab-case}.md`
- 예시:
  - `docs/impl/madupal/0427-rag-search-service.md`
  - `docs/impl/madupal/0428-context-cache.md`

작성자 폴더명은 GitHub ID 또는 팀에서 합의한 핸들을 사용하며, 소문자 kebab-case로 정규화합니다.

## 문서 템플릿

```markdown
---
generated-by: ai-draft
reviewed-by:
reviewed-at:
evidence: # 예: PR #5, 이슈 #12, 팀 논의 2026-04-26, ./gradlew test 결과
---

# {기능명}

> {한 줄 요약 - 뭘 만들었고 왜 만들었는지}

- 구현 일자: {날짜}
- 작성자: {author-kebab-case}
- 담당 파트: {담당 파트}
- 관련 테이블: {관련 테이블명 나열}

개발자가 사실관계와 근거를 검토한 뒤 직접 채웁니다.
```

## AI 초안 검토 규칙

- AI가 초안을 작성할 때는 `reviewed-by`, `reviewed-at`, `evidence`를 임의로 채우지 않습니다.
- 개발자가 사실관계와 근거를 검토한 뒤 `reviewed-by`, `reviewed-at`, `evidence`를 직접 채웁니다.
- `evidence`에는 PR, 이슈, 테스트 로그, 실험 결과, 공식 문서 등 확인 가능한 근거를 기록합니다.
- AI가 생성한 구현 기록은 PR 병합 전 `reviewed-by`, `reviewed-at`, `evidence`를 채워 검토 완료 상태로 전환합니다.
- 측정하지 않은 수치는 작성하지 않습니다.
- 수치가 필요한 경우 측정 방법, 비교 기준, 측정 일자를 함께 기록합니다.
- 근거가 없는 성능, 정확도, 비용 개선 표현은 작성하지 않습니다.
- 실험 결과는 실제 측정 데이터가 있을 때만 남기고, 데이터가 없으면 섹션 전체를 삭제합니다.

## 포트폴리오 메모 작성 가이드

- 기술 나열 금지: "pgvector를 사용했다" 같은 표현만으로 끝내지 않습니다.
- 문제 -> 해결 -> 결과 구조를 사용합니다.
- 측정된 숫자가 있으면 반드시 근거와 함께 포함합니다. 예: "토큰 60% 절감", "정확도 87%"
- 3문장 이내로 간결하게 작성합니다.

좋은 예시:

> 결정사항이 누적될수록 전체 주입 방식의 토큰 비용이 선형 증가하는 문제를 발견했다.
> RAG 구조를 도입해 질문과 유사한 컨텍스트만 선택적으로 주입하도록 설계했고,
> 토큰 사용량을 60% 줄이면서 답변 관련성은 향상됐다.

나쁜 예시:

> pgvector와 Spring Boot를 사용해 임베딩 기반 유사도 검색 기능을 구현했다.

## 사용 방법

구현이 끝난 뒤 AI 도구에 이렇게 요청합니다.

```text
방금 구현한 거 impl-record 템플릿대로 docs/impl/{author}/에 AI 초안으로 기록 만들어줘.
reviewed-by, reviewed-at, evidence는 비워둬.
```

또는 구현 요청할 때 미리 붙일 수 있습니다.

```text
decisions RAG 검색 서비스 구현하고, 끝나면 impl-record 템플릿대로 docs/impl/{author}/에 AI 초안 기록도 만들어줘.
reviewed-by, reviewed-at, evidence는 비워둬.
```
